# ALB・ネットワーク障害対応

## 1. 502 / 504 が多発した場合の切り分け
502（Bad Gateway）はターゲット側の応答不正やプロセスダウンが典型で、ターゲットグループのヘルスチェック状態を最初に確認する。
504（Gateway Timeout）はターゲットの応答遅延が原因のことが多く、ALB のアイドルタイムアウト（既定 60 秒）とアプリの処理時間を比較する。
どちらも ALB のアクセスログ（S3 保存）で該当時間帯の elb_status_code と target_status_code を突き合わせる。

## 2. ターゲットが unhealthy になる場合
ヘルスチェックのパス（/healthz）がアプリ側で 200 を返しているか、インスタンス内から curl で確認する。
セキュリティグループで、ALB からターゲットへのヘルスチェックポートが許可されているかを確認する。
デプロイ直後に落ちる場合は、ヘルスチェック猶予期間（HealthCheckGracePeriod）の不足を疑う。

## 3. 特定経路だけ通信できない場合
NACL とセキュリティグループの両方を確認する（NACL はステートレスなので戻りの通信も明示的な許可が必要）。
VPC フローログで REJECT レコードを検索し、どの ENI で通信が落ちているかを特定する。
