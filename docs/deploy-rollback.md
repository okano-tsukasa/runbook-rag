# デプロイ・ロールバック手順

## 1. 通常デプロイの流れ
デプロイは GitHub Actions から CodeDeploy 経由で行う。main ブランチへのマージで staging に自動デプロイされ、本番はワークフローの手動承認で実行する。
デプロイ前に Slack の #deploy チャンネルで宣言し、直前のリリースタグを控えておく。

## 2. デプロイ後の確認
ALB のターゲットグループで全ターゲットが healthy になっていることを確認する。
CloudWatch で 5xx エラー率とレイテンシ（p99）をデプロイ後 15 分間監視する。
5xx エラー率が 1% を超えた場合はロールバックを判断する。

## 3. ロールバック手順
CodeDeploy のコンソールから直前のリビジョンを指定して再デプロイする。
DB マイグレーションを伴うリリースの場合は、スキーマの後方互換が保たれているかを確認してからロールバックする。
互換がない場合はリードエンジニアと相談し、前方修正（fix forward）を優先する。

## 4. 緊急リリース（hotfix）
hotfix ブランチを main から切り、レビュー 1 名以上の承認後、staging を経由して本番へ出す。
緊急時でも staging を省略しない。省略が必要なほどの障害は Sev1 として扱い、インシデント対応フローに切り替える。
