# CloudWatch アラート初動対応 Runbook

## 1. アラート受信時の初動
CloudWatch アラートは Slack の #alerts チャンネルに ChatBot 経由で通知される。
通知を受けたら、まず対象のアカウントID・リージョン・リソース名・メトリクスを確認する。
5分以内に Slack のスレッドで「対応中」を表明し、一次切り分けを開始する。

## 2. 重大度（Severity）の判断
- Sev1: サービス全停止、または顧客影響が確認できる。即時エスカレーション。
- Sev2: 一部機能の劣化、リソース逼迫（CPU/メモリ/ディスクが閾値超過）。30分以内に対応着手。
- Sev3: 監視上の警告のみで顧客影響なし。営業時間内に対応。

## 3. EC2 CPU 使用率が高い場合
CPUUtilization が5分間 90% 超で発報する。
top や ps でプロセスを特定し、暴走プロセスや想定外のバッチが動いていないか確認する。
恒常的な高負荷ならインスタンスタイプ見直しを起票する。一過性ならスレッドに記録して経過観察。

## 4. ディスク使用率アラート
disk_used_percent が 85% 超で発報する。
df -h で逼迫しているマウントポイントを特定し、不要なログや一時ファイルを削除する。
/var/log が原因なら logrotate の設定を確認する。削除で回復しない場合は EBS ボリュームの拡張を検討する。

## 5. エスカレーション基準
一次切り分けで原因が特定できない、または Sev1 と判断した場合は、
オンコールのリードエンジニアへ電話と Slack メンションの両方で連絡する。
顧客連絡が必要な場合は、運用リードの承認を得てから行う。
