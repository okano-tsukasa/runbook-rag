# セキュリティインシデント対応 Runbook

## 1. IAM アクセスキー漏洩時の初動
GitHub 等への漏洩を検知したら、まず該当キーを IAM コンソールから無効化する（削除は調査が終わってから）。
CloudTrail で該当アクセスキーの直近 90 日の API 履歴を確認し、不審な操作（RunInstances、CreateUser、PutUserPolicy など）を洗い出す。
影響範囲が確定するまで、該当 IAM ユーザーの権限は一時的にすべて剥奪する。

## 2. 不審なリソース起動の検知
GuardDuty の Finding（CryptoCurrency 系、UnauthorizedAccess 系）は Slack の #security チャンネルへ通知される。
身に覚えのない EC2 / ECS リソースを発見した場合は、削除せずにまず隔離用セキュリティグループ（sg-quarantine）へ付け替え、フォレンジック用に EBS スナップショットを取得する。

## 3. 報告とエスカレーション
セキュリティインシデントは重大度にかかわらず、検知から 30 分以内にセキュリティ責任者へ第一報を入れる。
顧客データへのアクセス痕跡がある場合は Sev1 とし、法務への連絡も並行して行う。

## 4. 事後対応
インシデントの時系列（タイムライン）を作成し、5 営業日以内にポストモーテムを実施する。
再発防止として、漏洩経路に応じて git-secrets の導入や IAM ポリシーの最小権限化を起票する。
