# Runbook RAG — 運用・保守手順に質問できるローカルLLM RAG

社内の運用・保守手順（Runbook）に自然言語で質問でき、**回答の根拠となった手順を出典付きで提示する**ローカルLLM RAGアプリ。すべてローカル（Ollama）で動作し、社内ドキュメントを外部に送信しない。

> 題材として AWS 運用の Runbook を使っているが、`docs/` を差し替えるだけで
> 製造設備の保守マニュアルや業務手順書など、**任意のドメインに転用できる**構成にしている。
> なお `docs/` の手順書（9本）はすべて本プロジェクト用に書き下ろした架空のサンプルで、実在組織の情報は含まない。

## アーキテクチャ

```
質問
 └─▶ 埋め込み(bge-m3) ─▶ ベクトル検索(Chroma, top_k) ─▶ 参考手順チャンク
                                                          └─▶ LLM(Qwen3) ─▶ 回答 + 出典
```

| 層 | 採用 | 補足 |
|----|------|------|
| Chat LLM | Qwen3（Ollama） | 日本語性能が高く Apache 2.0 |
| 埋め込み | bge-m3 | 多言語・日本語に強い。Chat とは別モデル |
| ベクトルDB | Chroma | ローカル永続化。pgvector への置換も視野 |
| UI | Streamlit | 出典チャンクを画面に展開表示 |
| フレームワーク | LangChain | — |

## なぜこの構成か（設計判断）

RAG は「動かす」ことより「精度を作り込む」ことが本質なので、以下を意識した。

- **ハルシネーション抑制**: システムプロンプトで「参考手順に無いことは『記載がありません』と答える」を強制。回答の根拠を正式手順に限定している。
- **出典の明示**: 回答末尾と画面の両方で、根拠にしたチャンクの出典ファイルを表示。利用者が裏取りできる。
- **チャンク戦略**: 手順書は見出し単位で意味がまとまるため、`##` / `###` を優先セパレータにした `RecursiveCharacterTextSplitter` を採用。`chunk_size` は下記の評価で決定した。
- **埋め込みと Chat の分離**: 検索品質は埋め込みモデルでほぼ決まるため、Chat 用とは別に日本語に強い bge-m3 を選定した。

## 検索精度の評価

`eval/eval_set.json`（質問 × 期待出典 × 答えの根拠となる文言、全42問。手順書の語彙をそのまま使わない**言い換え質問10問**を含む）に対し、**hit-rate@4**・**hit-rate@1**・**MRR** を測定。判定は**チャンク単位**（正しい出典由来かつ根拠の文言を含むチャンクが top_k に入って初めてヒット）にしている。ファイル単位の判定では文書数が少ないうちは常にほぼ満点となり、chunk 設定の違いを測れないためである。`chunk_size` を変えて比較した結果：

| chunk_size | overlap | hit-rate@4 | hit-rate@1 | MRR |
|-----------:|--------:|-----------:|-----------:|----:|
| **300** | **50** | **1.00** | **0.90** | **0.95** |
| 600 | 100 | 0.98 | 0.83 | 0.90 |
| 1000 | 150 | 1.00 | 0.86 | 0.91 |

→ hit-rate@4 はほぼ飽和して差が出にくかったため、「正解チャンクを1位に出せたか」を見る **hit-rate@1** を評価に追加した。その結果 **chunk_size=300 が全指標で最良**だった。手順書は見出し単位で意味が完結するため、チャンクを小さくして「1チャンク≒1トピック」に近づけるほど、質問と無関係な文が埋め込みに混ざらず順位が上がったと考えている。この結果を `config.py` の既定値（300/50）に採用した。

再現コマンド：

```bash
python eval/evaluate.py
```

## セットアップ

```bash
# 1. Ollama でモデルを取得
ollama pull qwen3:8b
ollama pull bge-m3

# 2. 依存をインストール
pip install -r requirements.txt

# 3. docs/ をインデックス化
python ingest.py

# 4. UI 起動
streamlit run app.py
```

Ollama 推奨環境変数（デフォルトの context は 2048 と短く、RAG では文脈が切れやすい）：

```bash
export OLLAMA_CONTEXT_LENGTH=8192
export OLLAMA_FLASH_ATTENTION=1
```

## 今後の改善（ロードマップ）

- ハイブリッド検索（BM25 + ベクトル）とリランカーで精度向上
- ベクトルDBを Chroma → **pgvector** に置換（PostgreSQL の運用知識を活用）
- 回答の正誤も LLM-as-judge で自動評価し、評価を二層化
- 「手順を検索するだけ」から「複数手順をたどって実行計画を立てる」**Agentic RAG** への発展

## 必要スペック

ボトルネックは GPU の VRAM。デフォルト構成（`qwen3:8b` + `bge-m3`）で合計およそ 6〜7GB を使う。

| 構成 | GPU / VRAM | メモリ | 体感 |
|------|-----------|--------|------|
| 推奨 | NVIDIA VRAM 8GB 以上（RTX 3060 / 4060 クラス〜） | 16GB+ | 快適 |
| 軽量化時 | VRAM 4〜6GB | 16GB+ | `qwen3:4b` / `gemma3:4b`（約2.5GB）に変更で快適 |
| 最小 | GPU なし（CPU のみ） | 16GB+ | 動くが生成は遅い |

- **ディスク**: モデル約 6〜7GB ＋ Python 環境約 2GB ＝ 合計 10GB ほど。
- **軽量化**: `config.py` の `CHAT_MODEL` を `qwen3:4b` や `gemma3:4b` に替えるだけで、非力な PC でも現実的に動く。賢さは少し落ちるが題材としては十分。
- **Mac**: Apple Silicon でも動作（メモリ共有のため M 系 16GB あれば 8B 構成が快適）。
